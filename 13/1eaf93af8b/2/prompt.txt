2: 
3: This document details the architecture, UX patterns, and implementation guidelines for building a filtering system inspired by Linear’s issue filtering, tightly integrated with **React** + **TanStack Table** (using `@tanstack/react-table` v8) and **TanStack Virtual**. 7: ## 1. Overview
8: 
9: The filtering system allows users to narrow down tabular data by applying **any number of conditions** on any column. Two interaction modes are provided:
10: 
11: - **Quick Filter (keyboard)**: type directly in a filter bar (activated by a shortcut) to filter across a default “search” column, or use an inline chip syntax to target specific columns.

23:    - Keyboard‑first (typing in a “filter bar”, autocomplete for columns/operators/values).
24:    - Combobox‑first (mouse‑friendly, select column → operator → value).

98: ## 4. Filter Model & Data Types
99: 
100: ### 4.1 Filter Condition Shape (TypeScript)
101: 
102: ```ts
103: type FilterOperator =
104:   | "eq" | "neq"
105:   | "gt" | "gte" | "lt" | "lte"
106:   | "contains" | "notContains"
107:   | "startsWith" | "endsWith"
108:   | "isEmpty" | "isNotEmpty"
109:   | "in" | "notIn";
110: 
111: interface FilterCondition {
112:   id: string;          // unique identifier
113:   columnId: string;    // column key as defined in table
114:   operator: FilterOperator;
115:   value: unknown;      // string, number, boolean, Date, string[] (for in)
116: }
117: 
118: type FilterGroup = {
119:   id: string;
120:   type: "group";
121:   combinator: "and" | "or";
122:   conditions: (FilterCondition | FilterGroup)[];
123: };
124: 
125: type FilterState = FilterGroup; // root is always a group

188: ### 5.2 Converting FilterState to TanStack Filters
189: 
190: We cannot directly feed the hierarchical `FilterState` into TanStack’s flat `columnFilters`. Instead:
191: 
192: - **Global filter**: the filter bar’s “search all” text is passed to `globalFilter`.
193: - **Advanced filters**: a custom global filter function that recursively evaluates the `FilterState` tree. TanStack’s `filterFns` can be extended.
194: 
195: Option A: Use a single custom global filter function that takes the row and the `FilterState` as filter value:
196: 
197: ```tsx
198: const table = useReactTable({
199:   data,
200:   columns,
201:   state: { globalFilter: filterState },
202:   globalFilterFn: advancedFilterFn,
203:   // ...
204: });
205: ```
206: 
207: The `advancedFilterFn` receives `row`, `columnId` (unused), `filterValue: FilterGroup`. It evaluates the tree against each row.
208: 
209: Option B: Flatten the tree into an array of conditions, then use a custom `filterFn` on a pseudo‑column. This works but loses grouping logic if not careful.
210: 
211: **Recommended: Option A** – the global filter holds the `FilterState` object, and we implement a recursive evaluation function. This cleanly separates the filter UI state from TanStack’s column-level filtering.
212: 
213: ### 5.3 Recursive Evaluation Function
214: 
215: ```ts
216: function evaluateFilterGroup(
217:   row: Row<Data>,
218:   group: FilterGroup,
219:   getValue: (columnId: string) => unknown
220: ): boolean {
221:   const results = group.conditions.map((cond) => {
222:     if (cond.type === "group") {
223:       return evaluateFilterGroup(row, cond, getValue);
224:     }
225:     // Evaluate condition
226:     const cellValue = getValue(cond.columnId);
227:     return evaluateCondition(cellValue, cond);
228:   });
229: 
230:   return group.combinator === "and"
231:     ? results.every(Boolean)
232:     : results.some(Boolean);
233: }
234: ```
235: 
236: `evaluateCondition` applies the operator based on column type (e.g., `contains` for strings, `gt` for numbers/dates).
237:

239: 
240: - Use `useState` or a Zustand store for `filterState: FilterGroup`.
241: - Sync with URL search params (e.g., `?filters=...`) via a JSON.stringify/parse round‑trip (with compression if needed).
242: - When filter state changes, update the TanStack table’s `globalFilter` state.
243: 
244: ```tsx
245: const [filterState, setFilterState] = useState<FilterGroup>({
246:   id: "root",
247:   type: "group",
248:   combinator: "and",
249:   conditions: [],
250: });

258: 
259: TanStack Virtual expects a flat array of rows. Because we filter at the data level before rendering the table, virtualisation works naturally:
260: 
261: 1. Raw data → apply `filterFn` (which uses the `FilterState`) → filtered data array.

263: 3. The table exposes `table.getRowModel().rows`, which are already filtered and paginated if needed.
264: 4. Virtualise those rows using `useVirtualizer`.

360: ## 8. Performance Considerations with TanStack Virtual
361: 
362: - **Filter evaluation cost**: For client‑side filtering, the recursive `evaluateFilterGroup` is called for every row on every filter change. With virtualisation, only visible rows are rendered, but **filtering still runs on the entire dataset**. For large datasets (>10k rows), consider:
363:   - Memoising the filtered result with `useMemo` based on raw data and filter state.
364:   - Using Web Workers to evaluate filters off the main thread.
365:   - Moving filtering to the server (server‑side filtering) – the `filterState` is sent to the API.
366: - **Value suggestions**: For data‑driven value lists, pre‑compute distinct values once on data load, or fetch them lazily when a column is selected to avoid blocking the UI.
367: - **Avoid unnecessary re‑renders**: Use `React.memo` on condition rows and comboboxes. Keep filter state in a separate context provider (or Zustand) so that the table does not re‑render on every keystroke in the filter bar unless the committed filter state changes.

406:   value: unknown,

11: - **Quick Filter (keyboard)**: type directly in a filter bar (activated by a shortcut) to filter across a default “search” column, or use an inline chip syntax to target specific columns.
12: - **Advanced Filter Builder (combobox)**: a dedicated panel/modal where users compose conditions by selecting columns, operators, and values from dropdowns. Supports nested AND/OR groups.
13: 
14: The filter state is fully serialisable (e.g. in URL search params) and integrates seamlessly with TanStack Table’s client-side filtering and virtualisation.
15:

18: ## 2. Core Requirements
19: 
20: 1. **Combine any number of filters** – unlimited conditions, linked by AND/OR.
21: 2. **Nest groups** – (e.g., `(A AND B) OR (C AND D)`) for complex logic.
22: 3. **Two input modes**:
23:    - Keyboard‑first (typing in a “filter bar”, autocomplete for columns/operators/values).
24:    - Combobox‑first (mouse‑friendly, select column → operator → value).
25: 4. **Column‑aware** – each column exposes its data type, available operators, and value source (static list, fetched, or free‑form).
26: 5. **TanStack Table integration** – the filter state transforms into `columnFilters`, `globalFilter`, and custom `filterFns`.
27: 6. **Virtualisation friendly** – filtering happens on the **data level** before rows are passed to the virtualised table.
28: 7. **URL synchronisation** – filter state can be saved in query parameters for shareable views.
29: 8. **Accessibility** – full keyboard navigation, ARIA labels, screen‑reader support.

98: ## 4. Filter Model & Data Types
99: 
100: ### 4.1 Filter Condition Shape (TypeScript)
101: 
102: ```ts
103: type FilterOperator =
104:   | "eq" | "neq"
105:   | "gt" | "gte" | "lt" | "lte"
106:   | "contains" | "notContains"
107:   | "startsWith" | "endsWith"
108:   | "isEmpty" | "isNotEmpty"
109:   | "in" | "notIn";
110: 
111: interface FilterCondition {
112:   id: string;          // unique identifier
113:   columnId: string;    // column key as defined in table
114:   operator: FilterOperator;
115:   value: unknown;      // string, number, boolean, Date, string[] (for in)
116: }
117: 
118: type FilterGroup = {
119:   id: string;
120:   type: "group";
121:   combinator: "and" | "or";
122:   conditions: (FilterCondition | FilterGroup)[];
123: };
124: 
125: type FilterState = FilterGroup; // root is always a group
126: ```
127: 
128: ### 4.2 Column Metadata
129: 
130: Each column must define filter metadata via TanStack Table’s `meta`:
131: 
132: ```ts
133: interface ColumnFilterMeta {
134:   type: "string" | "number" | "date" | "boolean" | "enum";
135:   operators?: FilterOperator[]; // override default operators for type
136:   enumValues?: { label: string; value: string }[]; // static enum options
137:   valueSource?: "static" | "data-driven" | "async";
138:   // for async: function that returns options based on input string
139:   asyncSuggestions?: (query: string) => Promise<Option[]>;
140: }
141: ```
142: 
143: ### 4.3 Operators per Type (default)
144: 
145: | Type    | Operators                                                                 |
146: |---------|---------------------------------------------------------------------------|
147: | string  | eq, neq, contains, notContains, startsWith, endsWith, isEmpty, isNotEmpty |
148: | number  | eq, neq, gt, gte, lt, lte, isEmpty, isNotEmpty                            |
149: | date    | eq, neq, gt, gte, lt, lte, isEmpty, isNotEmpty                            |
150: | boolean | eq, neq                                                                   |
151: | enum    | eq, neq, in, notIn                                                        |
152: 
153: ### 4.4 Combining Filters
154: 
155: The UI builds a tree of `FilterGroup` nodes. At evaluation time (client‑side), the tree is recursively applied:
156: 
157: - `and` combinator: all children must match.
158: - `or` combinator: at least one child matches.

162: ## 5. Integration with TanStack Table
163: 
164: ### 5.1 Defining Columns with Meta
165: 
166: ```tsx
167: const columns = [
168:   columnHelper.accessor("status", {
169:     header: "Status",
170:     meta: {
171:       filter: {
172:         type: "enum",
173:         enumValues: [
174:           { label: "Open", value: "open" },
175:           { label: "Done", value: "done" },
176:         ],
177:       },
178:     },
179:   }),
180:   columnHelper.accessor("title", {
181:     header: "Title",
182:     meta: { filter: { type: "string" } },
183:   }),
184:   // ...
185: ];
186: ```
187: 
188: ### 5.2 Converting FilterState to TanStack Filters
189: 
190: We cannot directly feed the hierarchical `FilterState` into TanStack’s flat `columnFilters`. Instead:
191: 
192: - **Global filter**: the filter bar’s “search all” text is passed to `globalFilter`.
193: - **Advanced filters**: a custom global filter function that recursively evaluates the `FilterState` tree. TanStack’s `filterFns` can be extended.
194: 
195: Option A: Use a single custom global filter function that takes the row and the `FilterState` as filter value:
196: 
197: ```tsx
198: const table = useReactTable({
199:   data,
200:   columns,
201:   state: { globalFilter: filterState },
202:   globalFilterFn: advancedFilterFn,
203:   // ...
204: });
205: ```
206: 
207: The `advancedFilterFn` receives `row`, `columnId` (unused), `filterValue: FilterGroup`. It evaluates the tree against each row.
208: 
209: Option B: Flatten the tree into an array of conditions, then use a custom `filterFn` on a pseudo‑column. This works but loses grouping logic if not careful.
210: 
211: **Recommended: Option A** – the global filter holds the `FilterState` object, and we implement a recursive evaluation function. This cleanly separates the filter UI state from TanStack’s column-level filtering.
212: 
213: ### 5.3 Recursive Evaluation Function
214: 
215: ```ts
216: function evaluateFilterGroup(
217:   row: Row<Data>,
218:   group: FilterGroup,
219:   getValue: (columnId: string) => unknown
220: ): boolean {
221:   const results = group.conditions.map((cond) => {
222:     if (cond.type === "group") {
223:       return evaluateFilterGroup(row, cond, getValue);
224:     }
225:     // Evaluate condition
226:     const cellValue = getValue(cond.columnId);
227:     return evaluateCondition(cellValue, cond);
228:   });
229: 
230:   return group.combinator === "and"
231:     ? results.every(Boolean)
232:     : results.some(Boolean);
233: }
234: ```
235: 
236: `evaluateCondition` applies the operator based on column type (e.g., `contains` for strings, `gt` for numbers/dates).
237: 
238: ### 5.4 State Management
239: 
240: - Use `useState` or a Zustand store for `filterState: FilterGroup`.
241: - Sync with URL search params (e.g., `?filters=...`) via a JSON.stringify/parse round‑trip (with compression if needed).
242: - When filter state changes, update the TanStack table’s `globalFilter` state.
243: 
244: ```tsx
245: const [filterState, setFilterState] = useState<FilterGroup>({
246:   id: "root",
247:   type: "group",
248:   combinator: "and",
249:   conditions: [],
250: });
251: 
252: useEffect(() => {
253:   table.setGlobalFilter(filterState);
254: }, [filterState, table]);
255: ```
256: 
257: ### 5.5 Virtualisation Compatibility
258: 
259: TanStack Virtual expects a flat array of rows. Because we filter at the data level before rendering the table, virtualisation works naturally:
260: 
261: 1. Raw data → apply `filterFn` (which uses the `FilterState`) → filtered data array.
262: 2. Pass filtered data to `table` via `data` prop.
263: 3. The table exposes `table.getRowModel().rows`, which are already filtered and paginated if needed.
264: 4. Virtualise those rows using `useVirtualizer`.
265: 
266: No special treatment needed as long as filtering is applied to `data` before it enters the table.

297:

301: 
302: - Input field with `onKeyDown` to handle autocomplete.
303: - Manages a local string that is parsed into chips.
304: - Autocomplete uses a headless combobox library (e.g., Downshift, or a custom hook).
305: - On `Enter`, if a valid condition is formed, a chip is added to the quick‑filter list. The quick‑filter list is a flat array that is ANDed together – internally mapped to a `FilterGroup` with `and` combinator.
306: - Chips are removable via Backspace or `×` button.

314:

316: 
317: All comboboxes follow a consistent pattern:
318: 
319: - **Trigger**: button showing current selection or placeholder.
320: - **Popover**: searchable list of options.
321: - **Keyboard**: Arrow keys to navigate, Enter to select, Escape to close.
322: - **Async suggestions**: for data‑driven values, fetch distinct column values (or use pre‑fetched data) filtered by search query.
323: 
324: Implementation can be based on Radix UI’s Popover + Command (cmdk), or a headless library like `@tanstack/react-combobox` if available. For simplicity, a controlled `input` with a dropdown list works.
325: 
326: ### 6.3 State Machine for Keyboard Flow
327: 
328: Focus in filter bar → typing:
329: 
330: 1. **Column selection**: filter bar suggests column names from `table.getAllLeafColumns()` filtered by `meta.filter`.
331: 2. **Operator selection**: when `:` is typed or column selected, suggest operators based on column type.
332: 3. **Value input**: after operator is chosen (or automatically `:`), free‑form text input; suggestions based on `meta.filter.enumValues` or `asyncSuggestions`.
333: 4. **Commit**: pressing `Enter` (or `Tab`) creates the chip and clears the input for the next condition.

335: ### 6.4 Parsing and Serialisation
336: 
337: - The quick‑filter bar text is parsed into an array of `FilterCondition` using a simple regex or parser (handling quoted values for spaces, e.g., `status:"in progress"`).
338: - Advanced filter state is serialised to/from a compact JSON string stored in URL parameter `f`.
339: - The two sources (quick + advanced) are combined into one `FilterGroup` with `and` at root.

360: ## 8. Performance Considerations with TanStack Virtual
361: 
362: - **Filter evaluation cost**: For client‑side filtering, the recursive `evaluateFilterGroup` is called for every row on every filter change. With virtualisation, only visible rows are rendered, but **filtering still runs on the entire dataset**. For large datasets (>10k rows), consider:
363:   - Memoising the filtered result with `useMemo` based on raw data and filter state.
364:   - Using Web Workers to evaluate filters off the main thread.
365:   - Moving filtering to the server (server‑side filtering) – the `filterState` is sent to the API.
366: - **Value suggestions**: For data‑driven value lists, pre‑compute distinct values once on data load, or fetch them lazily when a column is selected to avoid blocking the UI.
367: - **Avoid unnecessary re‑renders**: Use `React.memo` on condition rows and comboboxes. Keep filter state in a separate context provider (or Zustand) so that the table does not re‑render on every keystroke in the filter bar unless the committed filter state changes.

373: ### 9.1 Custom Global Filter Function
374: 
375: ```ts
376: // filterUtils.ts
377: export const advancedFilterFn: FilterFn<Data> = (
378:   row,
379:   columnId,
380:   filterValue: FilterGroup
381: ) => {
382:   return evaluateFilterGroup(row, filterValue, (colId) => row.getValue(colId));
383: };
384: 
385: function evaluateFilterGroup(
386:   row: Row<Data>,
387:   group: FilterGroup,
388:   getValue: (colId: string) => unknown
389: ): boolean {
390:   for (const condition of group.conditions) {
391:     if (condition.type === "group") {
392:       const result = evaluateFilterGroup(row, condition, getValue);
393:       if (group.combinator === "or" && result) return true;
394:       if (group.combinator === "and" && !result) return false;
395:     } else {
396:       const cellValue = getValue(condition.columnId);
397:       const result = evaluateCondition(cellValue, condition);
398:       if (group.combinator === "or" && result) return true;
399:       if (group.combinator === "and" && !result) return false;
400:     }
401:   }
402:   return group.combinator === "and";
403: }
404: 
405: function evaluateCondition(
406:   value: unknown,
407:   cond: FilterCondition
408: ): boolean {
409:   // ... implement operator logic based on column meta type
410:   // use column metadata if needed (passed via condition or looked up)
411: }
412: ```

414: ### 9.2 Filter State Hook (Zustand example)
415: 
416: ```ts
417: import { create } from "zustand";
418: 
419: interface FilterStore {
420:   advancedFilter: FilterGroup;
421:   quickConditions: FilterCondition[];
422:   setAdvancedFilter: (group: FilterGroup) => void;
423:   addQuickCondition: (cond: FilterCondition) => void;
424:   removeQuickCondition: (id: string) => void;
425:   clearAll: () => void;
426:   // Computed: merge into a single FilterGroup
427:   mergedFilter: () => FilterGroup;
428: }
429: 
430: export const useFilterStore = create<FilterStore>((set, get) => ({
431:   advancedFilter: { id: "advanced", type: "group", combinator: "and", conditions: [] },
432:   quickConditions: [],
433:   // ... implementations
434:   mergedFilter: () => {
435:     const { quickConditions, advancedFilter } = get();
436:     // Root AND: quickConditions (as and-group) AND advancedFilter
437:     const quickGroup: FilterGroup = {
438:       id: "quick",
439:       type: "group",
440:       combinator: "and",
441:       conditions: quickConditions.map((c) => ({ ...c, type: "condition" })),
442:     };
443:     return {
444:       id: "root",
445:       type: "group",
446:       combinator: "and",
447:       conditions: [quickGroup, advancedFilter],
448:     };
449:   },
450: }));
451: ```
